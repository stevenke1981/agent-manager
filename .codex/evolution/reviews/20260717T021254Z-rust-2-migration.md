- Review time (UTC): `2026-07-17T02:12:54Z`

## Task summary

Agent Manager was migrated from Python/tkinter to a Rust 2024 `eframe`/`egui` crate while preserving the existing `agents/` corpus. The result includes safe Agent storage, validation, templates, evolution, OpenRouter support, importing, 13-tool installation/backup, a responsive GUI, and a read-only headless `--check` gate. The migration worktree contains no changes under `agents/`; the verified scan loaded 37 categories and 306 `SKILL.md` files.

## Corrections and review findings

The focused re-review did not stop at compile success. It caught several correctness and data-loss risks in destructive operations, delayed UI work, evolution fallback, consolidated installers, imports, and bulk diagnostics. Each was converted into an explicit invariant and a regression test.

| Failure | Cause | Fix and evidence |
|---|---|---|
| Delete could accept a path that was lexically under `agents/` but did not have the exact Agent shape, or could leave sibling assets behind. | Prefix-only reasoning is insufficient for recursive deletion; backing up only `SKILL.md` does not preserve an Agent directory. | `storage::validated_skill_parts` requires exactly `agents/<category>/<slug>/SKILL.md`, rejects separators/traversal, checks canonical direct ancestry, refuses symlinks during backup, copies the complete Agent directory, verifies the backup contains `SKILL.md`, then deletes. Covered by traversal/nonstandard-shape and complete-directory-backup tests. |
| A delete confirmation or AI result could act on whichever Agent happened to be selected when the delayed action completed. | The original target and editor version were not durable identities across an asynchronous UI interval. | Delete confirmation stores a `DeletePending.path` snapshot. AI work carries a `DocumentBinding { path, revision }`; a mismatched target or revision marks the result stale and disables apply. Covered by the delete snapshot and AI binding tests. |
| A background mutation or refresh could replace an unsaved editor buffer, and a stale editor could later overwrite newer disk content. | UI busy state alone did not distinguish read-only work from disk mutation, and save had no conflict state tied to the document revision. | Dirty state blocks refresh, selection, create/import/evolve/delete, and mutating task launch. Mutating tasks lock editor/save/new/delete/selection; completion reloads the bound disk document only when path/revision still match. Dirty or mismatched completion sets `save_conflict`, which blocks save until re-selection. Disconnected workers follow the same conflict-safe recovery. Covered by the mutating-busy test and binding tests. |
| Evolution could persist an invalid LLM rewrite, use the mutated candidate as fallback input, or write `.evolution.log` during dry run. | Validation, fallback provenance, and secondary side effects were not enforced as one transaction boundary. | API candidates are parsed and validated before save; when validation is required, remaining issues reject the write. Fallback reloads the original document from disk before skeleton repair. Both skeleton/API paths receive a persist flag, and dry run skips Agent writes, backups, and log append. Covered by invalid-rewrite, disk-original-fallback, and no-write dry-run tests. |
| Aider/Windsurf multi-Agent install could repeatedly read, back up, and rewrite one consolidated target, while repeated installs could duplicate content. | The single-Agent installer was called once per skill even though these tools share one file. | `install_skills` handles consolidated tools as one batch: read once, fold stable marker upserts in memory, and perform at most one backup/atomic write only when content changes. The 50-skill test runs the batch twice and requires one marker per skill and at most one backup. |
| Nested import sources with the same filename could collide and overwrite or skip one another. | Identity was based on filename stem rather than the source-relative path. | Import slugs derive from all relative path components; an in-batch target set detects remaining collisions and adds a deterministic path hash suffix. Covered by the stable unique nested-source test. |
| Malformed `SKILL.md` files could disappear silently from scans and give a false clean result. | Fallible loads were previously treated as filterable items rather than checked diagnostics. | `list_skills_checked` returns both loaded skills and path-specific diagnostics. GUI scan displays diagnostics, evolution refuses to run when any load fails, and CLI prints `LOAD ERROR` plus `load_errors`. The integration test requires nonzero status for malformed YAML. |
| Rapid backups could reuse a timestamp directory. | Wall-clock timestamps alone are not a uniqueness guarantee at batch speed. | Backup roots combine nanosecond timestamps, a process-local atomic counter, and exclusive directory creation with retry. Covered by the rapid-backup uniqueness test. |

## Reusable skill decision

These lessons are cross-cutting engineering safeguards already owned by the existing Codex Evolution self-review/verification workflow; a Rust-2.0- or Agent-Manager-specific skill would be a narrow duplicate. The evidence-backed procedures were instead recorded concisely in `.codex/evolution/LESSONS.md`.

## Remaining risks

- YAML serialization preserves data semantics but may normalize comments, indentation, quoting, or scalar presentation; exact concrete-syntax roundtrip is not promised.
- OpenRouter was not live-tested because no user credential was used; model/API behavior remains an external integration risk.
- Tool installers were verified with temporary/unit paths, not writes to each user's live tool directory; upstream format changes remain possible.
- GUI verification is compile/test plus a bounded process smoke, not full visual or interaction automation.
- The existing corpus intentionally fails strict `--check` because CRITICAL/HIGH validation findings remain; this is a data-remediation backlog, not a migration gate defect.
